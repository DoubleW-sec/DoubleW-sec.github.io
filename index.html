Arbitrary file Code Execution Vulnerability

一、漏洞实践实例

某后台管理网站a_templetex.php目的是接收浏览器端传入的POST参数filename和contents，通过file_put_contents函数将contents内容写到filename对应的文件中。可以看到执行file_put_contents函数之前未对filename和contents做任何检查和过滤。

若contents内容为可执行的恶意代码，则最终生成的filename为木马文件。

因此在复现过程中POST参数提交filename=test.php以及一句话木马content= ，可以看到页面显示成功，并在后台源码根目录下生成了木马文件testp.php，通过URL访问可执行此文件中的一句话木马。



二、漏洞原理分析


此漏洞的成因来自于两个方面。首先，file_puts_contents的参数filename以及content来源于浏览器客户端，并且对参数的内容并无检查和过滤，使得攻击者可以写入构造的恶意代码，造成任意代码执行；其次是PHP本身存在将字符串转为PHP可执行代码的函数（eval、preg_replace、assert等），通过这些函数能够将浏览器端传入的伪造POST字符串转换为可执行代码，生成可执行的木马文件。
因此根据漏洞成因，任意代码执行漏洞的利用分为两个方面。首先在未经过检查和过滤的POST参数中添加构造的代码字符串，并通过PHP函数将其转换为可执行的PHP代码，并能够生成相应的木马PHP文件；其次在浏览器客户端中通过URL访问该木马PHP文件，并传入需要的参数，完成代码执行。【若在PHP函数中直接写入无需参数的代码字符串，则后续在浏览器客户端中直接访问木马PHP文件即可】

三、漏洞修复方法
（1）对file_put_content的参数进行检查和过滤，可添加白名单或黑名单，禁止敏感字符串写入。
（2）设置用户对file_put_content函数的使用权限。

Arbitrary File Read Vulnerability

一、漏洞实践实例
某后台管理网站a_templete.php的目的是读取目标文件的编码格式、文件大小、修改日期等信息，然后返回包含相应内容的HTML页面。
其中读取方式为针对浏览器端提交的POST数据中的filename和dir信息合成目标文件完整路径，然后通过file_get_contents函数直接将目标文件内容转换为字符串来进行后续处理。
可以看到此处没有对输入的POST数据filename和dir没有做任何过滤或检查。

因此若知晓或可猜测此网站后台包含的敏感文件名及路径，则可直接通过URL访问得到敏感信息。

二、漏洞原理分析


根据对某后台系统任意文件读取漏洞的复现和实践，发现任意文件读取漏洞一般发生在网站文件读取功能中。
网站服务器端的php文件针对浏览器端传过来的filename等POST参数获得该文件在网站上的绝对路径，对文件进行读取并返回HTML页面。

在网站文件读取功能的实现过程中可以发现漏洞利用点。若访问URL中的filename被替换，那么这一功能实现流程也完全可以进行，若没有对其进行检查或者过滤，那当目标文件被替换为敏感文件时，就会造成敏感信息泄露，造成安全风险。

若发现网站存在任意文件读取漏洞并想要进行漏洞利用，那么需要找到敏感信息的所在位置和文件名称，自以为可以有方法：（1）已知网站源码或部分开放URL，可直接通过构造目标URL获得服务器版本、系统详情、数据库信息等；（2）未知网站源码或开放URL，可逐层猜测尝试敏感文件路径和名称，直到达到目的。
这是因为一般网站的文件访问链接和关键文件名称和位置具有一定规律性。从链接来看，一般形式为read.php?filename=/download.php?path等；从参数名来看，一般形式为&FilePath=/&readﬁle=/&data=等；从文件名来看，常见的敏感文件名为sql、passworda/pwd、shadow、keys等。

三、漏洞修复方法

（1） 对URL中的参数进行过滤或者检查，可设置用户访问白名单或通过正则表达式进行匹配。
（2） 过滤点(.)使用户在url中不能回溯上级目录
（3） php.ini 配置open_basedir限定文件访问范围
（4） 配置用户访问web服务器的权限。

sql

一、漏洞实例实践

某系统后台ajax.php的POST数据中，N_type参数可加入延时判断，因此判断存在注入点。

从源码中可以看到N_type参数是预设好的值，并且是执行SQL语句的一部分，能够提供恶意SQL语句执行的条件。因此可在N_type参数处构造SQL语句执行获取敏感信息。

因此可在N_type参数处使用Sqlmap构造SQL语句执行获取敏感信息。
首先构造注入ajax.php的注入POST数据。然后sqlmap -r命令构造payload进行sql注入，能够获取到敏感信息数据库信息及用户信息。


二、漏洞原理分析


SQL注入，就是通过把SQL命令插入到Web表单提交或输入域名或页面请求的查询字符串，最终达到欺骗服务器执行恶意的SQL命令。从定义中就可以总结出SQL注入存在的特征：（1）具有参数提交，例如？name=value；（2）与数据库有交互行为，涉及参数提交。造成SQL注入漏洞原因有两个：一个是没有对输入的数据进行过滤(过滤输入)，还有一个是没有对发送到数据库的数据进行转义(转义输出)。

检测是否存在SQL注入漏洞主要分为简单判断和特意伪造两个方面。简单判断是在URL中直接添加数字、字符或搜索进行检测，比如http://host/test.php?id=100 and 1=1或http://host/test.php?name=rainman ‘ and ‘1’=‘1。特意伪造是更改常见SQL语句达到正常执行命令的目的，比如万能密码（‘or 1=1 or’或admin’ or 1=1）或字符串连接（http://host/test.jsp?ProdName=B’||’ook）。

进行SQL注入漏洞利用时，可以先试探性注入，比如延时攻击（注入语句中加入睡眠时间，以此根据响应时间判断猜测正确与否）或盲注（判断用户名第一个字符是不是“a”）；也可以更改SQL语句进行，比如基于错误的注入技术（在SQL注入语句中加入执行函数语句）或外连攻击技术（在SQL注入语句中加入外连其他服务器的执行函数，使得连接到其他服务器）；还可以使用拼接SQL语句注入，比如联合查询SQL注入技术（UNION联合多个SQL语句）

三、漏洞修复方法
（1） 使用参数化查询方式进行SQL查询。
（2） 对输入参数进行检查和过滤。
（3） 页面不错误回显。
